# Favicon (issue #77)

## Contexte

Aucun favicon n'existait dans le projet. Le logo de la navbar est l'emoji 🩺 codé en dur (`qcm/templates/qcm/base.html:63`).

## Solution retenue

Favicon **SVG** embarquant ce même emoji (`<text>` dans un `<svg viewBox="0 0 64 64">`), plutôt qu'un `.ico` binaire — cohérent avec la philosophie "pas de build step" du projet (Bootstrap/HTMX via CDN). Créé à `qcm/static/qcm/img/favicon.svg`.

`django.contrib.staticfiles` était déjà installé sans `STATICFILES_DIRS` custom, donc ce fichier est automatiquement découvert par le finder par défaut (`AppDirectoriesFinder`) — aucun changement de settings nécessaire.

## Piège rencontré : `<head>` indépendants hors de `base.html`

Première itération : ajout de `{% load static %}` + `<link rel="icon">` uniquement dans `qcm/templates/qcm/base.html`. L'utilisateur ne voyait pas le favicon en testant — la page affichée était `/login/`, qui (comme 6 autres templates `registration/`) a son **propre `<head>` complet, sans `{% extends "qcm/base.html" %}`**.

Templates avec `<head>` indépendant (favicon ajouté séparément dans chacun) :
- `qcm/templates/registration/login.html`
- `qcm/templates/registration/inscription.html`
- `qcm/templates/registration/inscription_done.html`
- `qcm/templates/registration/password_reset_form.html`
- `qcm/templates/registration/password_reset_done.html`
- `qcm/templates/registration/password_reset_confirm.html`
- `qcm/templates/registration/password_reset_complete.html`

`password_change.html` et `password_change_done.html` étendent bien `qcm/base.html` — déjà couverts, pas touchés.

À retenir pour toute future modification du `<head>` (meta tags, scripts globaux, etc.) : vérifier systématiquement ces 7 templates `registration/`, pas seulement `base.html`.

## Piège rencontré : `AppDirectoriesFinder` et nouveau dossier `static/`

Avant ce fix, `qcm/static/` n'existait pas du tout dans le repo. Un `runserver` déjà lancé avant la création de ce dossier renvoie 404 sur tout fichier qu'il contient, même après modification du fichier : `AppDirectoriesFinder` répertorie les dossiers `static/` de chaque app **une fois au démarrage du process** (via `get_finders()`), pas à chaque requête. Un nouveau fichier dans un dossier déjà connu est servi à chaud (DEBUG=True relit le fichier à chaque requête), mais un dossier `static/` **tout neuf** nécessite un redémarrage du serveur de dev pour être détecté.

## Tests
